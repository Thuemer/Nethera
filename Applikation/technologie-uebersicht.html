Wie die Anwendung funktioniert

Vom Klick im Browser bis zur gespeicherten Regel in der Datenbank. So kann später ein Router-Service die Regeln anwenden.

1

Benutzer öffnet die App

Die Oberfläche läuft im Browser und zeigt Dashboard, Geräte, Gruppen, Presets und Zeitlimits.

HTMLGrundstruktur der Seiten
CSSAussehen und Responsive Design
JavaScriptInteraktion und Daten laden
Beispiel: Eine Gruppe wird im Formular umbenannt.
2

Login und Rechte

Keycloak kann den Benutzer anmelden. Danach kann das Backend wissen, wer gerade arbeitet.

KeycloakLogin-System
JWT Tokendigitaler Ausweis für API-Anfragen
Beispiel: Mein Account zeigt Login-Status und Einstellungen.
3

Frontend fragt Backend

Die App sendet und lädt Daten über REST-Schnittstellen vom Backend.

REST APIklare URLs für Daten
FetchJavaScript lädt Daten vom Server
JSONDatenformat zwischen Frontend und Backend
Beispiel: Zeitlimit für ein Tablet speichern.
4

Backend verarbeitet Daten

Quarkus nimmt die Anfrage entgegen, prüft sie und speichert passende Objekte.

Java 21Programmiersprache im Backend
QuarkusBackend-Framework
Hibernate ORMverbindet Java-Objekte mit Tabellen
Beispiel: Gerät wird einer Gruppe zugeordnet.
5

Datenbank speichert Regeln

PostgreSQL speichert Geräte, Gruppen, Blocklisten, Presets, Zeitlimits und Traffic-Daten.

PostgreSQLzentrale Datenbank
Docker Composestartet Backend, Datenbank und Keycloak lokal
Später kann ein Router-Service diese Regeln aus der DB lesen.
Kurz gesagt Der Benutzer stellt Regeln in der Web-App ein. Das Backend speichert alles strukturiert in PostgreSQL. Dadurch sind die Daten bereit, damit ein Router-Service sie später technisch umsetzen kann.
Wichtige gespeicherte Daten Alles kommt aus der Datenbank und kann dort weiterverarbeitet werden.
